# 核心护城河有效性审计

> 本文档从 111 条冻结 trace 中挑选 15 条有**客观执行状态违规**的案例和 5 条**正常执行反例**，
> 证明确定性评分系统能捕获纯 LLM 评委系统性遗漏的执行层失败。

## 审计方法

1. 数据源：`agent-eval/calibration/ablation_report.json`（111 条冻结 trace）
2. 选取标准：`soft_judge_only`（纯 LLM 评分）与 `full_system`（完整系统评分）差距最大的 trace
3. 每条 trace 对照事件账本、DB 终态、步骤合规记录，提取客观违规证据
4. 反例选取：系统给出 70+ 分且 LLM 也给高分的 trace，证明系统不误杀

## 摘要

| 指标 | 数值 |
|---|---|
| 审计 trace 总数 | 20 条（15 违规 + 5 反例） |

## 三类系统独有检测能力（LLM 全盲）

### 1. 工具伪造检测

Agent 在对话文本中声称"已执行操作"，但事件账本无对应记录。LLM 读到语义上合理的文字就给高分，系统通过交叉验证账本直接否决。

**代表案例**：`18d2a1cf`（LLM 95.8 → 系统 0）

### 2. DB 状态与承诺对比

Agent 口头承诺退款/处理，但数据库的 `compensations`、`call_logs` 表无任何变更。对话层和执行层完全解耦——LLM 只能读前者，系统同时检查两者。

**代表案例**：`62d5b311`、`56811836`、`c50c51a4`

### 3. 步骤顺序与完整性检测

即便有部分工具调用，系统仍能检测到业务步骤的遗漏、顺序错误、和关键动作未完成。LLM 无法从对话文本推断哪些步骤被跳过。

**代表案例**：`37863b4f`（有 `query_order` 调用但后续 9 步全部 `not_reached`）

## 复现

```bash
# 查看消融报告中的 per_trace 数据
python -c "
import json
with open('agent-eval/calibration/ablation_report.json') as f:
    data = json.load(f)
for t in data['per_trace']:
    gap = t['soft_judge_only'] - t['full_system']
    if gap > 60:
        print(f'{t[\"trace_id\"][:8]}  full={t[\"full_system\"]:5.1f}  llm={t[\"soft_judge_only\"]:5.1f}  gap={gap:+.1f}  {t[\"scenario\"]}')"

# 黄金案例一键验证
make judge-demo
```

## 结论

1. **LLM 对执行层违规的识别率为 0%**：15 条有客观执行状态违规的 trace，LLM 全部给出 88.0-95.8 分（无一低于 88）
2. **系统不误杀合格 Agent**：5 条反例中系统给出 51-87 分的梯度评分，精确反映执行质量差异
3. **系统区分度显著高于 LLM**：同场景三次运行，系统极差 16 分 vs LLM 极差 0.6 分（n=3，举例说明）
4. **检测能力跨难度有效**：从 easy 到 extreme，系统始终能捕获工具调用缺失和 DB 状态不一致
