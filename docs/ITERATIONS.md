## 2026-06-02 — 维度评分从一刀切到独立路由

**目标**：让 D1-D6 每个维度有独立的规则信号来源，而不是总分复制六遍。

**改了什么**：`scorer_outbound.py` 新增 `_compute_rule_dimensions()`，15+ 规则检查路由到对应维度。

**什么坏了**：两个独立审查 subagent 发现 D6 零证据虚高（合作场景无工具调用但给满分）、D3 违规分类错误、opening 为 None 时崩溃。

**学到什么**：写代码的 agent 不能审自己的代码——盲审 subagent 10 秒抓到主 agent 漏掉的问题。弱信号维度（D2/D3/D5）不能纯路由，需要混合 baseline。

**下一步**：Oracle 全量验证 → 确认维度区分度。

## 2026-05-25 — 人工标注暴露 rubric 缺陷

**目标**：用人工标注校准系统评分。

**改了什么**：48/123 条标注完成，二元一致率 93.8%。

**什么坏了**：标注者发现对话截断（只展示 3 轮）导致无法判断；维度题缺少参考信息；rubric 与场景不匹配（"被识别为机器人"在自报 AI 场景下无效）。

**学到什么**：标注工具的质量直接决定标注数据的质量——工具缺陷会系统性地污染金标数据。标注必须在代码冻结后做，否则白费。

**下一步**：修标注工具 → 补参考面板 → rubric 缺陷记入 LIMITATIONS.md。

## 2026-05-23 — Phase 4 安全强化：从声称覆盖到真实覆盖

**目标**：对齐 SafeToolBench 九维安全框架。

**改了什么**：D6 原子从 5 个扩充到 8 个；用户模拟器加 5 级自适应对抗；5 个新对抗场景。

**什么坏了**：`_update_adversarial_state()` 忘了去重，同一条拒绝重复计数导致策略过快升级；`dimensions_triggered` 只填了 6/9 维。

**学到什么**：声称九维覆盖 ≠ 代码九维覆盖。必须有测试验证每个维度至少被触发一次。

**下一步**：补 dim5/dim6/dim9 触发逻辑 + 去重防护。

## 2026-05-10 — 消融实验：证明系统存在的意义

**目标**：量化"规则层到底贡献了多少"。

**改了什么**：消融实验框架（5 种配置 × 111 条 trace），对比完整系统 vs 各组件缺失版本。

**什么坏了**：无——这是第一次用数据证明系统价值。

**学到什么**：LLM 独评 88.8% vs 完整系统 37.2%，差 51.6pp。这个数字成了整个项目的核心叙事。去掉 LLM 评委只 -0.3pp，证明 LLM 对总分几乎没贡献（但在软质量维度仍有价值）。

**下一步**：这个数据变成 README 第一段、路演开场白、所有文档的核心论据。
