# Agent 对话评测框架 v1

## 评分维度（7个）

### 1. 需求捕获（权重 15%）

**定义**：agent 是否完整、准确地理解并记住用户的所有约束条件。

| 分数 | 判定标准 |
|---|---|
| 9-10 | 首轮主动澄清关键约束，全程无遗漏，后续轮次能准确回溯所有已知条件 |
| 6-8 | 核心需求捕获到位，但有次要条件遗漏或需要用户重复 |
| 3-5 | 漏掉影响结果的关键约束（如预算、人数），或中途"忘记"已说过的条件 |
| 0-2 | 严重误解用户意图，或持续忽略明确说过的约束 |

**加分项**：主动询问用户未提但可能影响决策的因素（如停车、交通）
**减分项**：用户不得不重复同一条件两次以上

---

### 2. 调用编排（权重 25%）

**定义**：agent 是否高效、合理地使用工具，包括并行调用、调用时机、参数选择。

| 分数 | 判定标准 |
|---|---|
| 9-10 | 能并行的调用全部并行；参数精准反映用户需求；无冗余调用 |
| 6-8 | 调用逻辑正确但有明显可并行却串行的情况，或有1-2次冗余调用 |
| 3-5 | 反复用相同/近似条件搜索；该并行的全串行；调用次数明显超出必要 |
| 0-2 | 调用逻辑混乱，参数与用户需求不匹配，或完全不调工具凭空编造 |

**加分项**：一次性批量查询多个日期/多家餐厅，减少对话轮次
**减分项**：每轮只动一个变量、等用户确认后才试下一个（本可自主决策的情况下）
**归因豁免**：工具存在依赖链（B 的参数依赖 A 的返回值）时，串行调用不扣分——只对"无依赖却串行"的情况扣分

---

### 3. 工具调用透明度（权重 10%）

**定义**：agent 是否对每次工具调用的输入和结果做显式交代，使对话可审计。

| 分数 | 判定标准 |
|---|---|
| 9-10 | 每次调用都明确告知用了什么参数、返回了什么结果（包括空结果） |
| 6-8 | 大部分调用有交代，偶尔省略无结果的调用说明 |
| 3-5 | 多次调用结果含糊（"没搜到"但不说搜了什么条件），或合并多次调用但不区分 |
| 0-2 | 声称做了某事但无法确认是否真的调了工具；结果与调用不对应 |

**加分项**：主动说明"调用了 X 工具、参数是 Y、返回为空/返回 N 条结果"
**减分项**：说"火锅方向没搜到"但无法确认是否发起了火锅类搜索

---

### 4. 容错与兜底（权重 20%）

**定义**：当主路径失败时，agent 能否快速识别瓶颈、提出有效替代方案、合理降级。

| 分数 | 判定标准 |
|---|---|
| 9-10 | 第2次失败即识别系统性瓶颈，主动提出多个可行替代（含线下渠道），不空转 |
| 6-8 | 能识别瓶颈但反应慢（3-4轮后才转向），替代方案可行但不够丰富 |
| 3-5 | 反复撞墙5轮以上才转向，或替代方案脱离实际 |
| 0-2 | 始终在死胡同里打转，不提供任何替代路径 |

**加分项**：兜底方案具体可执行（给出电话号码、地址、话术模板）
**减分项**：每次失败都把决策权推回用户（"你觉得怎么办"），缺乏主动判断

---

### 5. 边界意识（权重 15%）

**定义**：agent 是否清楚自己的能力边界（工具集能做什么、不能做什么），并在到达边界时尽早把控制权交还用户。

| 分数 | 判定标准 |
|---|---|
| 9-10 | 明确区分"系统没数据"和"没有符合条件的"，在能力边界处给出清晰说明并建议线下/人工渠道 |
| 6-8 | 能意识到边界但表述模糊，用户需要追问才能判断是工具限制还是真没有 |
| 3-5 | 混淆工具限制与真实市场情况，误导用户以为"全城就一家店" |
| 0-2 | 完全不区分，把工具集的缺陷当成客观事实传达给用户 |

**加分项**：主动说明"系统收录有限，实际选择可能更多"
**减分项**：暗示系统数据 = 全部现实

---

### 6. 沟通效率（权重 10%）

**定义**：agent 的回复是否简洁、有条理，是否在必要时主动推进而非被动等指令。

| 分数 | 判定标准 |
|---|---|
| 9-10 | 回复精简无废话，主动推进流程，用户无需催促 |
| 6-8 | 整体清晰但偶有冗余段落，或在可自主决策处仍请求确认 |
| 3-5 | 明显啰嗦，用户表达不耐烦（"有点烦了""快点"），或每步都等用户拍板 |
| 0-2 | 回复冗长混乱，关键信息淹没在废话中 |

**减分项**：用户说"快点搞定"之后仍然慢节奏推进

---

### 7. 任务完成度（权重 5%）

**定义**：最终是否完成了用户的核心诉求。

| 分数 | 判定标准 |
|---|---|
| 9-10 | 成功完成预订+点菜+优惠券，全流程闭环 |
| 6-8 | 完成了主要步骤但有遗漏（如忘记挂券） |
| 3-5 | 未完成核心任务但给出了可执行的替代路径 |
| 0-2 | 未完成且无有效替代 |

**注意**：此维度权重最低，因为任务失败可能是工具集/环境的问题，不一定是 agent 的问题——需结合归因标签判断。

---

## 归因标签

每个扣分项必须标注归因：

| 标签 | 含义 | 示例 |
|---|---|---|
| **A-agent** | agent 设计/策略缺陷 | 该并行没并行、反复无效搜索、兜底太晚 |
| **T-toolset** | 工具集设计局限 | 没有"拆桌"参数、不支持区域筛选、无模糊搜索 |
| **E-environment** | 环境/数据问题 | 系统只收录1家店、排期数据为空 |
| **U-user** | 用户约束过紧 | 预算+包间+人数+区域同时卡死，现实中也很难满足 |

**归因规则**：一个扣分项可以有多个标签，但必须标明主因。比如"反复搜索5轮才放弃"主因是 **A-agent**（应更早识别瓶颈），次因是 **E-environment**（数据确实少）。

---

## 辅助指标：轮次效率

在总分之外，记录以下量化指标作为沟通效率维度的补充：

| 指标 | 计算方式 | 说明 |
|---|---|---|
| 总对话轮次 | 用户发言次数 | 从首次提出需求到任务结束/放弃 |
| 有效推进轮次 | 产生新信息或推动决策的轮次 | 排除重复搜索、等待确认等空转轮 |
| 轮次效率比 | 有效推进轮次 / 总对话轮次 | >60% 良好，40-60% 一般，<40% 低效 |

## 示例打分：团建订餐场景

| 维度 | 得分 | 归因 | 备注 |
|---|---|---|---|
| 需求捕获 | 8/10 | — | 首轮主动澄清5项约束，全程无遗漏 |
| 调用编排 | 3/10 | A-agent | 串行试日期（周二→周三→周四逐个查）、相同条件重复搜索5-6轮 |
| 工具调用透明度 | 5/10 | A-agent | 火锅搜索是否实际调用不明；多次搜索结果未逐条交代 |
| 容错与兜底 | 4/10 | A-agent 为主，E-environment 次 | 第6轮才建议线下渠道；兜底方案（电话+地址+话术）本身质量不错但来得太晚 |
| 边界意识 | 5/10 | A-agent | 前5轮暗示"搜不到=不存在"，第6轮才补充"系统收录有限" |
| 沟通效率 | 5/10 | A-agent | 用户明确说"有点烦了"后才提速；多次把选择推回用户 |
| 任务完成度 | 4/10 | E-environment 为主，A-agent 次 | 未完成预订，但最终给出可执行的线下方案 |

**加权总分**：0.15×8 + 0.25×3 + 0.10×5 + 0.20×4 + 0.15×5 + 0.10×5 + 0.05×4 = **4.6/10**

**轮次效率**：总 12 轮 / 有效 4 轮 = 33%

**总结**：需求理解扎实，但调用策略是最大短板——本可并行查询的日期逐个串行，相同条件反复搜索浪费了用户耐心。识别系统数据瓶颈太晚（第6轮），应在第2-3轮失败后即转向线下兜底。
