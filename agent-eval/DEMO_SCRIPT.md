# 路演脚本 — 8 分钟

> 核心叙事：复杂外呼指令在执行过程中会失败——走错分支、跳过确认、越权承诺、伪造工具结果。我们把 Markdown 任务指令编译成可审计的模拟实验室。

---

## 0:00 – 1:00 问题引入

**台词：**

> 各位评委好。我们先看一个数字：外呼场景的指令通常有 6-8 个步骤、3-4 个条件分支、若干禁止行为。
>
> 业务方真正想知道的不是"这个模型得了 78 分"——一个分数没有用。他们需要知道：**哪一步失败了、在什么用户行为下失败的、有什么证据、怎么修。**
>
> 现有评测方案要么是人工听录音打分——一通电话 15 分钟；要么是端到端跑个通过率——你只知道"没过"，不知道为什么。
>
> 我们做的事情是：**把一份 Markdown 任务指令，编译成一个可审计的模拟实验室。** 10 秒出报告，精确到步骤、轮次和根因。

**屏幕：** 展示一个典型的外呼任务指令文档截图（Markdown 格式，包含 Role / Task / Call Flow / Constraints 段落），旁边用红色标注"6 个步骤 / 3 个分支 / 4 条约束 / 2 个禁止行为"。

**动作：** 无操作，纯叙述，PPT 或静态页面。

---

## 1:00 – 2:00 粘贴指令 → 编译预览

**台词：**

> 我们的系统接受任意 Markdown 格式的任务指令。现在我粘贴一份真实的美团外呼指令进去。
>
> 大家看右边的编译预览面板——指令被自动解析成了结构化的评测计划：5 个执行步骤，每步有完成条件；2 个条件分支——"用户同意"走确认流程，"用户拒绝"走挽回流程；3 条禁止行为——不能说"公司规定"、不能透露内部系统信息、不能承诺超出权限的补偿。
>
> 这不是手工配置的——编译器从 Markdown 的结构里自动提取步骤、分支、约束、FAQ 知识点。你换一份指令，评测计划也跟着变。

**屏幕：** Streamlit 界面。左侧 sidebar 选择"粘贴 Markdown 指令"，文本框里有完整指令；右侧"指令编译预览"面板展开，清晰显示：
- 基本信息（任务 / 开场白 / 回复长度限制 / 最大轮次）
- 指令步骤列表（每步带 step_id、指令内容、分支箭头）
- 禁止行为列表（带严重程度标签）
- 知识点 / FAQ 列表

**动作：** 把预先准备的 Markdown 指令粘贴到文本框 → 页面自动渲染编译预览。

---

## 2:00 – 4:00 配合型用户跑一遍

**台词：**

> 现在我们跑第一轮评测。用户模拟器设定为"配合型"——对方会正常接听、确认信息、同意方案。我点"运行评测"。
>
> *(等待 ~10 秒)*
>
> 出来了。86 分。大家看步骤执行详情——5 步全绿，全部 completed。开场白通过，结束语通过，通话结果"已确认"与数据库交叉验证一致。Rubric 六个维度都在 4 分以上。
>
> 这是基线。一个合格的 Agent 在理想条件下应该拿到这个分数。但问题是——**你的用户不会永远配合你。** 接下来才是重点。

**屏幕：** Streamlit 界面完整展示：
1. 对话轨迹面板——聊天气泡形式，Agent（绿色）和被叫方（灰色）交替，工具调用显示为绿色成功标签
2. 评估报告面板——综合得分 86/100，各项指标：硬指标 95%、步骤合规 100%、分支准确 100%
3. 步骤详情——每步前面有绿色对勾

**动作：**
1. 左侧 sidebar 确认"快速模式"已勾选
2. 点击"运行评测"按钮
3. 等待加载动画结束
4. 滚动展示对话轨迹 → 评估报告 → 步骤详情

## 4:00 – 6:00 切换对抗用户（核心 2 分钟）

**台词：**

> 现在我切换到对抗场景。这个场景里，用户接起来就说"我很忙别烦我"，第三轮直接问"你是不是机器人"，第五轮情绪爆发——"你们外卖都送成这样了还打电话骚扰"。
>
> 跑一下。
>
> *(等待 ~10 秒)*
>
> 分数暴跌到 47 分。我们看发生了什么。
>
> **步骤详情**：step_3 信息确认——跳过了。step_4 方案推荐——失败。Agent 在用户说"我很忙"之后直接跳到了结束语，中间两步全没做。
>
> 看**禁止行为违规**：第 4 轮 Agent 说了"根据公司规定"，这是禁止话术，被检测到了。
>
> 现在看最关键的部分——**诊断分析**。系统不只是告诉你"47 分"，它告诉你：
>
> - **首次偏差**：第 3 轮。预期执行"确认用户身份"，实际直接跳到了结束语
> - **失败模式**：过早终止通话 + 步骤跳过 + 情绪处理失当
> - **根因**：模型在检测到用户负面情绪后，将"结束通话"误判为最优策略，跳过了中间的安抚和确认步骤
> - **修复建议**：Harness 步骤门控——必要步骤未完成时拦截结束意图；情绪检测——用户负面情绪时注入"先共情再处理"指令
>
> 这就是为什么我们说"一个分数没有用"。47 分背后是 3 个独立的失败点，每个都有精确的定位和可操作的修复建议。

**屏幕：**
1. 对话轨迹面板——第 2 轮用户消息高亮"我很忙别烦我"，第 4 轮 Agent 消息标红（禁止词检测）
2. 评估报告——47/100，步骤合规 40%，禁止行为违规 1 次
3. 步骤详情——step_1 和 step_2 绿色对勾，step_3 黄色跳过标记，step_4 红色失败标记，step_5 灰色未到达
4. 诊断分析面板——清晰展示偏差点、失败模式、根因、修复建议

**动作：**
1. 左侧 sidebar 切换到"选择预置场景" → 选择 `delay_notify_difficult`
2. 点击"运行评测"
3. 结果出来后，先指步骤详情（红绿对比），再指诊断面板
4. 用鼠标在"修复建议"处停留 3 秒，让评委看清

---

## 6:00 – 7:00 Harness 拦截演示

**台词：**

> 刚才的诊断建议开启 Harness 安全围栏。我们现在勾选"启用 Harness"，用同一个对抗场景再跑一次。
>
> 大家看对话轨迹里多了蓝色的系统消息。第 3 轮 Agent 试图直接结束通话——被拦截了，系统注入提醒："通话还不能结束，缺少必要步骤：确认用户身份"。第 4 轮 Agent 用了禁止词——被阻断，要求重新生成。
>
> 这就是我们系统的核心设计理念：**不信模型自报，只认可观测事件。** 5 条契约写在代码里：来源可信——打分只看工具实际返回值，不看 Agent 说"我已经退款了"；执行顺序——拦截必须发生在工具执行之前；结果严格——失败、超时、被拦截一律不算成功。
>
> Harness 介入后，同一个对抗场景分数从 47 回到了 71。不是模型变强了，是安全围栏补偿了模型的弱点。

**屏幕：**
1. 对话轨迹——穿插蓝色系统消息（拦截 + 步骤提醒 + 禁止词阻断）
2. 评估报告——71/100（对比刚才的 47）
3. Harness 干预统计：拦截 2 次、步骤注入 3 次、禁止词过滤 1 次

**动作：**
1. 勾选"启用 Harness（安全围栏）"
2. 点击"运行评测"
3. 结果出来后，指出蓝色系统消息 → 指出分数对比（47 → 71）

## 7:00 – 8:00 覆盖矩阵 + 商业价值

**台词：**

> 最后说一下系统的可靠性和商业价值。
>
> 可靠性方面：21 项对抗测试覆盖了所有已知的作弊手段——Agent 谎报工具结果、工具调用失败但日志记成功、跳过确认直接操作、禁止词变体绕过。11 项契约测试保证五条系统不变量。每次提交代码自动跑门禁，过不了不让合。
>
> 覆盖矩阵告诉你评测的盲区在哪：6 个场景、4 种用户画像交叉——哪些步骤被测过、哪些分支从没走过、哪些约束从没被触发。不是"跑了 100 遍就够了"，而是"覆盖了多少评测面"。
>
> 商业价值一句话：**上线前的 QA 门禁。** 一份新的外呼指令从 Markdown 到评测报告，10 秒。不需要人工听录音，不需要手工配置评测用例。粘贴进去，跑，看哪里断了，修，再跑。
>
> 谢谢各位评委。

**屏幕：** 总结页，三列布局：
- 左列：5 条系统契约图标（来源可信 / 执行顺序 / 结果严格 / 可审计 / 事件顺序）
- 中列：覆盖矩阵示意（步骤覆盖 100% / 分支覆盖 75% / 约束覆盖 80% / 故障覆盖 100%）
- 右列：关键数字——21 项对抗测试、11 项契约测试、6 个评测场景、10 秒出报告

**动作：** 切到总结 PPT 或静态页面，不做操作。
