## Recommendation: build a lightweight professional workflow, not a giant methodology stack

For a solo 2–3 day Python hackathon, the best toolchain is:

**Install now**
1. **Deterministic Python quality gate**: `ruff`, `pytest`, `pytest-cov`, `hypothesis`, `mypy` or `pyright`, `pre-commit`, plus a Claude Code `PreToolUse` hook that blocks `git commit` when the gate fails.
2. **Feature Dev plugin**: use it for multi-file features, scoring/harness changes, and architectural refactors.
3. **Pyright LSP plugin**: high ROI in Claude Code because it gives Python type/code intelligence inside the agent workflow.
4. **Optional PR Review Toolkit or Code Review plugin**: useful before major commits, but not needed on every small change.
5. **Context7 only if you rely on fast-moving third-party APIs**. For mostly pure Python evaluator logic, it is optional.

**Skip or defer**
1. **Superpowers full enforcement** for this hackathon. Borrow its discipline, especially TDD for critical evaluator logic, but do not introduce strict global “tests-first-or-delete” behavior right before a deadline.
2. **Skill-based pre-commit quality guard** as an enforcement mechanism. Use deterministic hooks instead.
3. **Large lifecycle suites** like `claude-code-skills` unless this becomes a longer-term project.
4. **Full semantic-release machinery**. Use Conventional Commits plus a manually curated `CHANGELOG.md`/`ITERATIONS.md` for the presentation story.

The key principle: **Claude skills/plugins guide behavior; hooks, tests, and CI enforce behavior**. Claude’s own docs distinguish guidance from enforcement: `CLAUDE.md` and skills are context/instructions, while hooks can block tool calls before they run. `PreToolUse` fires before tool execution, and exit code `2` blocks the tool call, which is exactly the enforcement primitive you need. ([Claude](https://code.claude.com/docs/en/memory))

| Phase | Best Claude Code tool | What it produces | Hackathon configuration |
|---|---|---|---|
| 1. Project Bootstrap | `/init`, `/memory`, `/project-automation-buildout`, deterministic scripts | `CLAUDE.md`, tooling, hooks, CI, repo docs | One-time setup; keep it lean |

## E. Add deterministic quality gate

Create `scripts/quality_gate.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail

COVERAGE_MIN="${COVERAGE_MIN:-60}"

echo "== Ruff format check =="
uv run ruff format --check .

echo "== Ruff lint =="
uv run ruff check .

echo "== Mypy =="
uv run mypy src

echo "== Security scan =="
uv run bandit -q -r src || {
  echo "Bandit found issues. If false positive, document and suppress narrowly."
  exit 1
}

echo "== Fast tests with coverage =="
uv run pytest -q -m "not slow and not adversarial" \
  --cov=src \
  --cov-branch \
  --cov-fail-under="${COVERAGE_MIN}"
```

Then:


Set `COVERAGE_MIN` to whatever passes today, then ratchet upward. For a new project, start at 70. For your current project, do not block progress with an unrealistic threshold; the more important gate is that **contract tests for the three discovered bugs must exist and pass**.

## G. Add Claude Code `PreToolUse` commit blocker

This is the right way to use Claude Code hooks: block Claude’s `git commit` Bash call before it executes if the quality gate fails. Claude Code hooks can match `Bash` tool calls, and exit code `2` blocks the call. ([Claude](https://code.claude.com/docs/en/hooks))

Create `.claude/hooks/block_bad_commit.py`:

```python
#!/usr/bin/env python3
from __future__ import annotations

import json
import re
import subprocess
import sys
from pathlib import Path

COMMIT_PATTERN = re.compile(r"(^|[;&|])\s*git\s+commit\b")


def main() -> int:
    payload = json.load(sys.stdin)
    tool_input = payload.get("tool_input") or {}
    command = tool_input.get("command") or ""

    if not COMMIT_PATTERN.search(command):
        return 0

    gate = Path("scripts/quality_gate.sh")
    if not gate.exists():
        print(
            "Blocked git commit: scripts/quality_gate.sh is missing.",
            file=sys.stderr,
        )
        return 2

    result = subprocess.run(
        ["bash", str(gate)],
        text=True,
        capture_output=True,
        check=False,
    )

    if result.returncode != 0:
        print("Blocked git commit: quality gate failed.\n", file=sys.stderr)
        print(result.stdout, file=sys.stderr)
        print(result.stderr, file=sys.stderr)
        return 2

    return 0


if __name__ == "__main__":
    raise SystemExit(main())
```

Then:

```bash
chmod +x .claude/hooks/block_bad_commit.py
```

Add `.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "python3 .claude/hooks/block_bad_commit.py"
          }
        ]
      }
    ]
  }
}
```

This is better than a “pre-commit quality guard skill” because it is deterministic. The skill can remind Claude; the hook can stop Claude.

# 3. Testing strategy: when to test first vs when to code first

Your bugs were not lint failures. They were **missing invariant tests**. The tool that catches them is not Ruff; it is executable contracts.

## Use test-first for these

Always write the failing test before code when touching:

- Scoring semantics.
- Harness execution order.
- Tool blocking/permission logic.
- Success/failure classification.
- Anything involving untrusted model claims.
- Any bug found by an auditor.
- Any behavior you will mention in the hackathon presentation.

Minimum regression tests for your three bugs:

```text
tests/contracts/test_execution_order_contract.py
  - test_blocking_check_happens_before_tool_side_effect

tests/contracts/test_scorer_source_of_truth_contract.py
  - test_model_self_reported_success_does_not_count_without_observed_success

tests/contracts/test_failed_operations_contract.py
  - test_failed_tool_operation_is_not_scored_as_success
  - test_blocked_operation_is_not_scored_as_success
  - test_exception_operation_is_not_scored_as_success
```

## Code-first is acceptable for these

- README edits.
- Changelog cleanup.
- Logging message wording.
- Throwaway experiment scripts.
- Demo formatting.
- Non-critical CLI ergonomics.

Even then, run the quality gate before commit.

## Test types and when to use them

| Test type | Use when | Example |
|---|---|---|
| Unit | Pure function behavior | `score_operation(event) -> failed` |
| Contract | Cross-component invariant | self-report never becomes evidence |
| Integration | Harness + scorer together | simulated multi-turn dialogue with tool events |
| Property | Many generated variants | all failed/blocked statuses score as non-success |
| Adversarial | Gaming attempts | model says “I succeeded” after blocked tool |
| Promptfoo red team | LLM behavior/prompt surface changes | prompt injection or metric gaming attempts |

Hypothesis is especially useful for the third bug: generate many operation statuses and assert that only observed success statuses count as success. Promptfoo is useful for adversarial LLM evals and red-team scans around prompt, agent, and RAG-style behaviors. ([Hypothesis](https://hypothesis.readthedocs.io/en/latest/quickstart.html?utm_source=chatgpt.com))

# 4. Review workflow: which review tool to use when

Use reviews in this order:

## `/simplify`: after tests pass, before final review

Use when the code works but may be messy, duplicated, over-engineered, or inefficient. `/simplify` can modify recently changed files, so run it before the final correctness/security review. Claude Code describes it as a skill that reviews recent files for reuse, quality, and efficiency issues, then fixes them. ([Claude](https://code.claude.com/docs/en/commands))

Use:

```text

```

## `/review`: before committing meaningful changes

Use for ordinary correctness review. It is read-oriented and good for catching logic issues in the diff.

```text

```

## `/security-review`: whenever a trust boundary is involved

Use for:

- untrusted model output,
- shell/tool execution,
- file paths,
- external APIs,
- secrets,
- sandboxing,
- prompt injection,
- eval gaming.

Claude Code’s `/security-review` analyzes pending branch changes for injection, auth issues, and data exposure. ([Claude](https://code.claude.com/docs/en/commands))

```text
/security-review
```

## adversarial-review: before declaring evaluator changes done

Use for scorer/harness changes. Ask it to assume the model is trying to game the metric.

Good prompt:

```text
Run adversarial review on the scorer and harness changes. Assume the evaluated model is malicious and tries to obtain success credit without performing the required operation. Focus on self-reported success, failed tool calls, blocked actions, event ordering, and missing audit trails.
```

Use Promptfoo when the prompt/dialogue/eval target changes, or before the final presentation. Its red-team workflow is designed to generate and run adversarial test cases against a target, then report results. ## PR Review Toolkit: use selectively

Use the specific agents after larger diffs:

# 6. What would have caught your three actual bugs?

| Bug | What catches it | Tooling role |
|---|---|---|
| Harness blocks after tool executes | Contract test with side-effect sentinel | PreToolUse pattern reminds you what “before execution” means |
| Scorer trusts self-reported data | Source-of-truth contract + adversarial test | adversarial-review and `/security-review` help if contract is documented |
| Failed operations counted as success | Property test over operation statuses | Hypothesis + contract tests |

Static typing might help model the difference between `ObservedSuccess`, `ModelClaimedSuccess`, `FailedOperation`, and `BlockedOperation`, but only executable tests enforce the semantics.

A good data model would make invalid scoring harder:

```python
from dataclasses import dataclass
from enum import StrEnum


class ObservedStatus(StrEnum):
    SUCCESS = "success"
    FAILED = "failed"
    BLOCKED = "blocked"
    TIMED_OUT = "timed_out"
    EXCEPTION = "exception"


@dataclass(frozen=True)
class ObservedOperation:
    operation_id: str
    requested_action: str
    status: ObservedStatus
    evidence_ref: str


@dataclass(frozen=True)
class ModelClaim:
    operation_id: str
    claimed_success: bool
    text: str


def counts_as_success(operation: ObservedOperation) -> bool:
    return operation.status is ObservedStatus.SUCCESS
```

The architectural move is to make `ModelClaim` impossible to pass into `counts_as_success`.

# 7. Where to be strict vs where to cut corners

## Be strict

- Scorer/harness correctness.
- Evaluation contracts.
- Test-first for audited bugs.
- Pre-commit and Claude `PreToolUse` commit blocker.
- Traceability of every success/failure decision.
- Commit history and changelog.

## Be pragmatic

- Coverage threshold. Start at current passing level, then ratchet.
- Type checking. Strict for `src/evaluator`, `src/scorer`, `src/harness`; relaxed for tests/scripts.
- Feature Dev. Use for multi-file or architectural changes, not one-line fixes.
- Promptfoo/adversarial runs. Run before milestones, not every commit.

## Cut corners

- Do not build a perfect plugin ecosystem.
- Do not enforce full TDD globally.
- Do not run mutation testing on the whole repo.
- Do not over-document every minor edit.
- Do not install giant lifecycle suites during the hackathon.

# Final tool decision matrix

| Tool/plugin | Decision | Why |
|---|---:|---|
| `feature-dev` | **Install** | Best match for missing architecture/design/review process |
| Claude `PreToolUse` hook | **Use** | Deterministic, blocks before bad commit/tool action |
| Actual `pre-commit` | **Use** | Protects commits outside Claude Code too |
| `pre-commit-quality-guard` skill | **Skip** | Prompt guidance, not enforcement |
| `pre-commit-review` skill | **Skip for every commit** | Too slow/noisy; use targeted reviews instead |
| Superpowers | **Defer** | Excellent discipline, too globally strict for 2–3 day deadline |
| Context7 | **Optional** | Useful for changing external APIs; not core to evaluator logic |
| Pyright LSP | **Install** | High ROI Claude Code Python intelligence |
| PR Review Toolkit | **Optional install** | Useful for test/error/silent-failure reviews |
| Promptfoo red team | **Use at milestones** | Best for adversarial dialogue/eval behavior |
| adversarial-review | **Use on scorer/harness** | Best solo substitute for a hostile reviewer |
| git-cliff | **Optional near final** | Good commit-to-changelog generator, but manual narrative still needed |

The minimal professional workflow is: **`/init` + CLAUDE.md + CONTRACTS.md + Feature Dev for risky work + deterministic pre-commit/PreToolUse gates + contract/adversarial tests + staged reviews + iteration log.** That gives you the highest protection per setup hour without turning the hackathon into process theater.
